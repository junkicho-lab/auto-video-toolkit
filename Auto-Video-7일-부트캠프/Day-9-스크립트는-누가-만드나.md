# Day 9 (심화) — 실제 결과물을 내는 `scripts/`는 누가, 어떻게 만드나

> [!success] 이 노트의 목표
> "손발" 역할인 `scripts/*.py`가 **누가, 언제, 어떻게** 만들어지는지 이해합니다. AI Native 설계에서 가장 헷갈리는 "에이전트가 스크립트를 만드는가?"라는 질문에 답합니다.

선행: [[Day-8-에이전트-하네스와-AI-Native]], [[Day-3-에이전트-시스템]] · ⏱️ 예상 60분 · 형태: 심화 개념

---

## 한 줄 답

> **스크립트는 사람(개발자)이 — 보통 AI 코딩 도구의 도움을 받아 — "미리" 작성해 저장소에 넣어둡니다.**
> 런타임에 에이전트(LLM)가 즉석에서 만들어내는 게 **아닙니다.**

---

## 1. 왜 "미리" 작성하나? (절대 즉석 생성 X)

> [!important] 손발은 "고정"되어야 한다
> 이미지 저장, 영상 합성, API 호출 같은 일은 **매번 똑같이, 정확히** 돌아야 합니다. LLM이 매번 새로 코드를 짜면 결과가 들쭉날쭉하고 위험합니다(파일 삭제 등).
> → 그래서 스크립트는 **결정적(deterministic) 코드로 한 번 잘 짜서 박제**해두고, LLM은 그걸 **호출만** 합니다.

| | 에이전트(.md) | 스크립트(.py) |
|---|---|---|
| 작성 시점 | 미리 (지시서) | **미리 (코드)** |
| 런타임에 바뀌나? | 내용 고정, 판단은 매번 | **절대 안 바뀜** |
| 누가 실행 결정 | LLM | 사람/오케스트레이터가 호출 |
| 성격 | 창의·판단 | 정확·반복 |

---

## 2. 누가 작성하나?

**개발자(사람)** 가 주체. 단, 요즘은 **AI 코딩 어시스턴트와 함께** 짭니다:

- 이 프로젝트의 원본 스크립트 → **원작자가 작성** (PDF 명세서에 전체 소스 포함)
- 실제로는 **Claude Code 같은 AI와 페어로** 작성하는 경우가 대부분
- **이번 부트캠프의 우리** → `generate_videos.py`·`produce.py`의 버그를 찾아 고침 = "사람이 AI와 함께 작성·유지보수"의 실제 모습

> [!tip] 빌드타임 vs 런타임 (중요한 구분)
> - **빌드타임(만들 때)**: 사람/AI가 스크립트를 작성 → 저장소에 커밋. (PDF 부트스트랩처럼 "Claude Code가 명세 보고 스크립트 생성"도 여기 해당)
> - **런타임(돌릴 때)**: 스크립트는 **이미 있는 고정 도구**. 에이전트는 파라미터만 정해서 호출.

---

## 3. 어떻게 작성하나? (스크립트 공통 골격)

이 프로젝트의 모든 스크립트는 **거의 같은 5단 구조**입니다. `generate_videos.py` 예:

```python
# ① 환경 로드 — config/api-keys.env에서 키 읽기
def load_env(): ...

# ② 입력 파싱 — 프롬프트 파일(.md)에서 씬별 데이터 추출
def parse_video_prompts(path): ...

# ③ 실제 작업 — 외부 API/라이브러리 호출 (provider별)
def veo_generate(prompt, out_path): ...     # Google veo
def kling_generate(...): ...                # Kling
# → PROVIDERS = {"veo": veo_generate, ...}

# ④ CLI 인터페이스 — argparse로 옵션 받기
def main():
    parser.add_argument("--project", ...)
    parser.add_argument("--provider", ...)

# ⑤ 루프 — 각 씬마다 생성 → 저장 → 성공/실패 집계
for s in prompts:
    fn(s["prompt"], out)
```

> [!example] 패턴이 핵심
> 모든 스크립트(`generate_images`, `produce`, `compositor`...)가 **이 골격을 공유**합니다. 하나를 이해하면 나머지도 읽힙니다. AI와 함께 짤 때도 "이 패턴대로 새 provider 추가해줘" 식으로 확장합니다.

### 스크립트가 하는 일의 본질
```
입력(프롬프트/설정) → 외부 도구 호출(API/라이브러리) → 파일 저장
```
- `generate_images.py` → ima2(GPT Image 2) 호출 → PNG 저장
- `generate_videos.py` → veo/kling API 호출 → MP4 저장
- `produce.py` → ElevenLabs API 호출 → WAV 저장
- `compositor.py` → MoviePy 라이브러리 → final.mp4

---

## 4. 에이전트는 스크립트와 어떻게 만나나

```
scene-director (에이전트/LLM)
   │  "이 씬은 veo로, 8초, 이런 프롬프트"  ← 판단
   ▼
오케스트레이터 (하네스)
   │  결정된 파라미터로
   ▼
generate_videos.py (스크립트/고정 도구)
   │  실제 veo API 호출
   ▼
scene-01/video.mp4  ← 결과물
```

> [!important] 한 문장 정리
> **에이전트(LLM)는 "무엇을 어떤 값으로"를 정하고, 스크립트(사람이 미리 짠 코드)는 "그 값으로 실제 실행"합니다.** 스크립트 자체는 LLM이 만들지 않습니다.

---

## 5. 우리가 이번에 한 게 바로 그 "작성/유지보수"

| 스크립트 | 한 일 | 작성 방식 |
|---|---|---|
| `generate_videos.py` | 키 파싱 버그 수정 + 폴더 생성 추가 | 사람이 지시 → AI가 코드 수정 → 검증 |
| `produce.py` | 같은 키 파싱 버그 수정 | 〃 |
| `autovideo/*.py` | 두뇌(하네스) 신규 작성 | 〃 |

→ "스크립트는 사람+AI가 미리 작성하고, 발견되는 문제를 고쳐가며 유지보수한다"의 실제 과정.

---

## 요약

```
누가:  개발자(사람) + AI 코딩 도구(Claude Code 등)
언제:  미리(빌드타임) — 런타임에 LLM이 즉석 생성 X
어떻게: 5단 골격(환경로드→입력파싱→API호출→CLI→루프)을 따라
        결정적 코드로 작성 후 저장소에 커밋
역할:  LLM이 "값"을 정하면, 스크립트가 그 값으로 "실행"
```

---

## ✅ 이해 점검

1. 스크립트는 런타임에 LLM이 만드나?
2. 스크립트를 "미리, 고정"으로 두는 이유는?
3. 스크립트 공통 5단 골격은?
4. "빌드타임 작성"과 "런타임 호출"의 차이는?
5. 에이전트와 스크립트의 역할 분담을 한 문장으로?

> [!success]- 정답 펼치기
> 1. 아니다. 사람(+AI 도구)이 미리 작성해 커밋해 둔다.
> 2. 정확·반복·안전 — 매번 똑같이 돌아야 하므로 결정적 코드로 박제.
> 3. 환경로드 → 입력파싱 → API/라이브러리 호출 → CLI(argparse) → 씬 루프.
> 4. 빌드타임=코드를 작성/커밋, 런타임=이미 있는 스크립트를 파라미터로 호출.
> 5. LLM은 "무엇을 어떤 값으로"를 정하고, 스크립트는 "그 값으로 실제 실행"한다.
