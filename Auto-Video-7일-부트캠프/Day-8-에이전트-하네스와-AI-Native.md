# Day 8 (심화) — 에이전트 하네스와 AI Native 구축

> [!success] 이 노트의 목표
> "에이전트 하네스(harness)가 뭔지 → 이 앱에서 어떻게 세팅돼 있는지 → AI Native하게 앱을 짓는 과정"을 초보자 눈높이로 이해합니다. 이 노트는 `autovideo/` CLI(Claude Code 없이 독립 실행되는 두뇌)를 직접 만들며 정리한 내용입니다.

선행: [[Day-3-에이전트-시스템]], [[Day-7-합성과-완주]] · ⏱️ 예상 90분 · 형태: 심화 개념

---

## 0. 먼저, "하네스(harness)"가 뭔가요?

> [!note] 비유
> **하네스 = 말(馬)에게 채우는 마구(안장·고삐·재갈).** 말(=LLM)은 힘이 세지만 혼자선 일을 못 합니다. 방향을 주고, 짐을 싣고, 멈추게 하는 **장치 일체**가 하네스입니다.

AI에서 하네스 = **"프롬프트 한 장 + 모델 하나"를 실제로 일하는 에이전트로 만들어주는 주변 코드 전부**입니다.

하네스가 하는 일 6가지:
1. **입력 준비** — 어떤 파일/정보를 읽어서 줄지
2. **프롬프트 조립** — 지시서 + 컨텍스트를 합쳐 모델에 넣을 문장 만들기
3. **모델 호출** — 어떤 모델로, 단발인지 대화인지
4. **출력 파싱** — 모델이 뱉은 텍스트에서 원하는 부분(YAML 등) 뽑기
5. **회복력** — 실패하면 재시도, 에러 안내
6. **사람 개입** — 체크포인트(승인/수정/중단), 상태 저장

---

## 1. 에이전트 "하나"의 해부

이 앱의 모든 에이전트(`agents/*.md`)는 똑같은 6칸 구조입니다:

```
## 역할 정의   ← "너는 OO 전문가다"        (정체성)
## 입력        ← "이 파일들을 읽어라"        (하네스가 채워줌)
## 출력        ← "이 파일로 저장하라"        (하네스가 받아냄)
## 진행 방식    ← "이렇게 질문/판단하라"       (행동 규칙)
## 품질 기준    ← "이 조건을 만족하라"        (검증 기준)
## 저장 지시    ← "Write로 저장, 다음 안내"   (다음 단계 연결)
```

> [!important] 핵심 통찰
> **에이전트 = 그냥 잘 쓰여진 텍스트(.md)** 입니다. 마법이 없습니다. 이 텍스트를 **하네스가 읽어서 모델에 먹이고, 결과를 받아 저장**할 때 비로소 "에이전트가 일한다"가 됩니다.

---

## 2. 이 앱의 하네스 = 만든 3개 부품

`autovideo/`에 만든 게 정확히 이 하네스입니다. 부품별 역할 매핑:

| 부품(파일) | 하네스 역할 | 비유 |
|---|---|---|
| `llm_client.py` | 모델 호출 + 재시도 + 모델선택 | **엔진** (말의 근육) |
| `agent_runner.py` | .md 로드 + 프롬프트 조립 + 출력 파싱 | **고삐·재갈** (방향 전달) |
| `orchestrator.py` | 순서·체크포인트·상태·스크립트 호출 | **마부** (전체 지휘) |
| `cli.py` | 사람이 명령 내리는 입구 | **마부에게 거는 말** |
| `scripts/*.py` | 실제 이미지/영상/음성 생성 | **수레·도구** (실제 짐) |

흐름:

```
당신 (cli.py)
   │  autovideo brand
   ▼
orchestrator (마부)  ── 순서 정하고, 체크포인트 관리
   │
   ├─ agent_runner: agents/brand-architect.md 읽기
   │                + brand 컨텍스트 합치기 → 프롬프트 완성
   │
   ├─ llm_client: Gemini 호출 (실패 시 재시도)
   │
   ├─ agent_runner: 응답에서 YAML 추출
   │
   └─ orchestrator: 미리보기 → [승인?] → brand/character.yaml 저장
```

## 3. 에이전트별 "하네스 세팅" (실제 값)

하네스는 에이전트마다 **다르게 세팅**됩니다.

### Phase 1 (브랜드) — 전부 "대화형 or 퀵"
| 에이전트 | 모델 성격 | 입력(읽기) | 출력 | 호출 방식 | 체크포인트 |
|---|---|---|---|---|---|
| brand-architect | 사고형 | 사용자 답변 | character.yaml | 대화형 / 퀵 1회 | 승인·수정·재생성 |
| narrative-designer | 사고형 | character.yaml | narrative.yaml | 〃 | 〃 |
| topic-strategist | 사고형 | character+narrative | topics.yaml | 〃 | 〃 |
| benchmark-analyst | 실행형 | 앞 3개 | benchmark.yaml | 〃 | 〃 |

### Phase 2 (콘텐츠) — "단발 생성 + 스크립트 호출"
| 에이전트 | 입력 | 출력 | 호출 방식 | 비고 |
|---|---|---|---|---|
| researcher | 주제 | research.md | 단발 1회 | LLM |
| scriptwriter | brand+research | script.md | 단발 1회 | LLM |
| scene-director | script | image/video-prompts | 단발 1회 | LLM |
| image-generator | prompts | scene-NN.png | **스크립트 실행** | ima2 (LLM 아님) |
| video-generator | prompts | scene-NN.mp4 | **스크립트 실행** | veo/kling… |
| voice-producer | script | scene-NN.wav | **스크립트 실행** | ElevenLabs |
| compositor | 전부 | final.mp4 | **스크립트 실행** | MoviePy |

> [!tip] 두 종류의 하네스
> - **"판단형" 하네스** (researcher~scene-director) → LLM을 부른다 (`llm_client`)
> - **"실행형" 하네스** (image~compositor) → Python 스크립트를 부른다 (`subprocess`)
>
> 같은 오케스트레이터가 **상대에 따라 다른 손**을 씁니다. 이게 "두뇌 vs 손발" 분리의 실제 모습입니다.

### 하네스 "공통 옵션" (직접 추가한 것)
| 옵션 | 의미 | 어디서 |
|---|---|---|
| 모델 선택 | `--model` / config / 기본값(gemini-2.5-flash) | `llm_client.model_for()` |
| 재시도 | 503/혼잡 → 2·4·8초 백오프 | `llm_client._retry()` |
| 할당량 처리 | 429 → 즉시 안내(재시도 X) | `llm_client._error_hint()` |
| 퀵 모드 | 호출 1회로 압축 | `agent_runner.build_brand_oneshot()` |
| 체크포인트 | 승인/수정/재생성/중단 | `orchestrator.checkpoint()` |

## 4. "AI Native 구축" 과정 = 6가지 원칙

AI Native하게 짓는다 = **"LLM을 부품으로 끼우는 게 아니라, LLM을 중심에 두고 설계한다."**

### ① 프롬프트를 "코드처럼" 다룬다
에이전트 = `.md` 파일(버전관리·재사용·리뷰 가능). 로직을 코드에 숨기지 않고 **읽을 수 있는 지시서**로 분리.

### ② 두뇌(LLM)와 손발(결정적 코드)을 분리한다
- 창의·판단이 필요한 건 LLM (대본·연출)
- 정확·반복이 필요한 건 일반 코드 (이미지 저장, 영상 합성)
- → LLM에 "파일 저장"을 시키지 않고, 코드에 "창작"을 시키지 않는다.

### ③ 같은 프롬프트를 여러 런타임에서 재사용한다
같은 `agents/*.md`를 **Claude Code도 쓰고, autovideo CLI도 쓴다.** 하네스만 갈아끼우면 됨. (멀티 플랫폼의 비밀)

### ④ 사람을 루프에 넣는다 (Human-in-the-loop)
단계마다 체크포인트로 **승인/수정**. AI가 혼자 끝까지 달리지 않게 — 품질·비용·신뢰의 핵심.

### ⑤ 실패를 가정하고 설계한다
LLM/네트워크는 자주 실패 → **재시도·할당량 안내·진행상황 저장(재개 가능)**. 우리가 겪은 503/429/인코딩/폴더 버그가 전부 여기 해당.

### ⑥ 모델을 작업에 맞게 고른다
사고형엔 강한 모델, 실행형엔 빠른 모델. 비용·품질 균형.

## 5. 실제로 거친 구축 여정 (= AI Native는 "반복")

AI Native 앱은 한 번에 안 됩니다. 한 사이클이 교과서적 예시였습니다:

```
설계 → 만들기 → 돌려보기 → 실패 → 원인분석 → 수정 → 재시도
```

| 단계 | 무엇 | 배운 점 |
|---|---|---|
| 두뇌 코어 작성 | llm_client/agent_runner/orchestrator | 하네스 3분할 |
| 첫 실행 | 404 (가짜 모델명) | 설정의 현실성 검증 필요 |
| 모델 고침 | 429 (무료 할당량) | 호출 수 줄이는 퀵 모드 |
| veo 시도 | 인코딩 에러 | 키 파싱 버그(인라인 주석) |
| 다시 | 폴더 없음 에러 | 저장 전 디렉토리 생성 |
| 자막 | veo가 글자를 영상에 구움 | "영상엔 글자 X, 자막은 SRT" 설계 원칙 |

> [!important] 진짜 교훈
> **버그 하나하나가 "AI Native 설계 원칙"을 가르쳐줬습니다.** 실패 → 원인 → 원칙 내재화. 이게 AI 앱을 짓는 실제 과정입니다. 깔끔한 한 방이 아니라, 현실의 마찰을 하네스로 흡수하는 반복.

3. "판단형 하네스"와 "실행형 하네스"는 각각 무엇을 부르나?
4. AI Native 원칙 중 "두뇌 vs 손발 분리"가 뜻하는 것은?
5. 같은 프롬프트를 Claude Code와 CLI가 함께 쓸 수 있는 이유는?

> [!success]- 정답 펼치기
> 1. 입력준비·프롬프트조립·모델호출·출력파싱·회복력(재시도)·사람개입(체크포인트/상태).
> 2. .md=지시서(무엇을), 코드=실행/검증/저장(어떻게).
> 3. 판단형→LLM(llm_client), 실행형→Python 스크립트(subprocess).
> 4. 창의·판단은 LLM, 정확·반복은 결정적 코드로 나눠 서로의 약점을 안 건드림.
> 5. 프롬프트가 .md로 분리돼 있어 하네스(런타임)만 갈아끼우면 되기 때문.
